Consumers & Consumer Groups

A consumer reads events from Kafka topics. Consumers work together in consumer groups to scale reading and provide fault tolerance.

Consumer Basics

Properties props = new Properties();
props.put("bootstrap.servers",    "localhost:9092");
props.put("group.id",             "order-processor");
props.put("key.deserializer",     "org.apache.kafka.common.serialization.StringDeserializer");
props.put("value.deserializer",   "org.apache.kafka.common.serialization.StringDeserializer");
props.put("auto.offset.reset",    "earliest");

KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);
consumer.subscribe(Arrays.asList("orders"));

while (true) {
  ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
  for (ConsumerRecord<String, String> record : records) {
    System.out.printf("offset=%d, key=%s, value=%s%n",
      record.offset(), record.key(), record.value());
  }
}

Consumer Groups

Consumers with the same group.id belong to the same consumer group. Each partition is assigned to exactly one consumer in the group — this is how Kafka scales consumption. 

# Example: Topic "orders" has 4 partitions
# Consumer group "order-processor" has 3 members
#
# Partition 0 → Consumer A
# Partition 1 → Consumer B
# Partition 2 → Consumer C
# Partition 3 → Consumer A  (one consumer handles multiple partitions)
Note If you have more consumers than partitions in a group, the extra consumers sit idle. Aim for consumers ≤ partitions in a group.

Offset Management

Kafka tracks which events each consumer group has read using offsets. Offsets are committed periodically so consumers can resume from where they left off after a restart.

Auto-commit (default)

props.put("enable.auto.commit", "true");
props.put("auto.commit.interval.ms", "5000");

Offsets are committed automatically every 5 seconds. Simple but can cause duplicate processing on restart.

Manual commit

props.put("enable.auto.commit", "false");

// After processing a batch:
consumer.commitSync();   // Blocks until committed
// or
consumer.commitAsync();  // Non-blocking

Rebalancing

When a consumer joins or leaves a group, a rebalance occurs: partitions are reassigned across the group. During a rebalance, no consuming happens.

Two rebalance protocols:

  • Eager (old) — all consumers stop, partitions revoked, then reassigned. Stop-the-world.
  • Cooperative (recommended) — consumers keep their partitions and only reassign a subset. Smoother.
props.put("partition.assignment.strategy",
  "org.apache.kafka.clients.consumer.CooperativeStickyAssignor");

Consumer Settings to Know

Setting Default What it does
auto.offset.reset latest Where to start when no committed offset exists
max.poll.records 500 Max events per poll() call
session.timeout.ms 45000 Max time before consumer is declared dead
max.poll.interval.ms 300000 Max time between poll() calls before rebalance

Exactly-Once Semantics

For end-to-end exactly-once processing, combine:

  1. Idempotent producer (enable.idempotence=true)
  2. Transactional producer — wrap produce + consume in a transaction
  3. isolation.level=read_committed on the consumer
Tip For most use cases, at-least-once (auto-commit + idempotent producer) + deduplication in your sink is simpler than full exactly-once. Only go transactional when you truly need it.

Key Takeaway

Consumer groups let you scale reading horizontally. Tune max.poll.records and session.timeout.ms for your processing speed. Use cooperative rebalancing for smoother operation.
