Kafka Connect New

Kafka Connect is a framework for streaming data between Apache Kafka and external systems — databases, key-value stores, search indexes, file systems, and more. Instead of writing custom producer/consumer code for every data source, you use pre-built connectors that handle the plumbing.

Why it matters Most Kafka deployments use Connect to ingest or export data. It's the standard way to integrate Kafka with the rest of your infrastructure.

Core Concepts

Connector

A connector is a piece of reusable code (a plugin) that knows how to talk to a specific system. Examples: JDBC connector for databases, S3 connector for object storage, Elasticsearch connector for search indexing.

Source vs Sink

  • Source connector — pulls data from an external system into Kafka. E.g., read rows from PostgreSQL and write them to a topic.
  • Sink connector — pushes data from Kafka to an external system. E.g., take messages from a topic and index them in Elasticsearch.

Worker

A Kafka Connect worker is a JVM process that runs connectors. Workers can run in two modes:

  • Standalone — single process, good for development
  • Distributed — cluster of workers with automatic rebalancing and fault tolerance (production)

Example: Postgres → Kafka with JDBC Source

Here's how to stream data from a PostgreSQL table into a Kafka topic:

# Connector config (JSON)
{
  "name": "postgres-orders",
  "config": {
    "connector.class": "io.confluent.connect.jdbc.JdbcSourceConnector",
    "connection.url": "jdbc:postgresql://localhost:5432/mydb",
    "connection.user": "kafka",
    "connection.password": "password",
    "table.whitelist": "orders",
    "mode": "incrementing",
    "incrementing.column.name": "id",
    "topic.prefix": "pg-",
    "poll.interval.ms": 5000
  }
}
Note The JDBC connector polls the database on an interval. It's not real-time streaming from the DB — for that, look at Change Data Capture (CDC) tools like Debezium.

Using the Connect REST API

Connect workers expose a REST API on port 8083 by default:

# List connectors
curl -s http://localhost:8083/connectors | jq .

# Create a connector
curl -s -X POST http://localhost:8083/connectors \
  -H "Content-Type: application/json" \
  -d @connector-config.json

# Check connector status
curl -s http://localhost:8083/connectors/postgres-orders/status | jq .

# Delete a connector
curl -s -X DELETE http://localhost:8083/connectors/postgres-orders

Key Connectors to Know

Connector Type Use Case
JDBC Source/Sink Both Relational databases (PostgreSQL, MySQL, Oracle)
Debezium Source (CDC) Change Data Capture from DB transaction logs
S3 Sink Sink Dump Kafka topics to S3 as Parquet/JSON/Avro
Elasticsearch Sink Sink Index Kafka messages for search/analytics
FileStream Source/Sink Both Read/write files (useful for testing)

Running Locally

Kafka ships with Connect bundled. To start a standalone worker:

# Start a standalone worker with a source connector
connect-standalone.sh config/connect-standalone.properties \
  config/connect-file-source.properties

For distributed mode (recommended for anything beyond testing):

# Start distributed worker
connect-distributed.sh config/connect-distributed.properties
Pro tip For local development with Docker, use the Confluent images: confluentinc/cp-kafka-connect. They include common connectors and can be configured via environment variables.

Connector Lifecycle

Connectors have a clear state machine:

  1. UNASSIGNED — created but not yet assigned to a worker
  2. RUNNING — actively processing data
  3. PAUSED — stopped processing but keeps configuration
  4. FAILED — encountered an error (check logs + status)

Pause/resume via the REST API:

# Pause a connector
curl -X PUT http://localhost:8083/connectors/postgres-orders/pause

# Resume
curl -X PUT http://localhost:8083/connectors/postgres-orders/resume
