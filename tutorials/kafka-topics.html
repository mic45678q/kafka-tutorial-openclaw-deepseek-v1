Topics & Partitions Core

Topics and partitions are the backbone of Kafka's storage model. Understanding how they work is essential to designing good Kafka architectures.

Topics

A topic is a logical channel to which producers publish events and from which consumers subscribe. Topics are multi-subscriber — many consumers can read from the same topic independently. 

# Create a topic with 3 partitions and replication factor 2
kafka-topics.sh --bootstrap-server localhost:9092 \
  --create --topic orders \
  --partitions 3 \
  --replication-factor 2

Partitions

Each topic is split into partitions. Partitioning is what gives Kafka its horizontal scalability:

  • Each partition is an ordered, immutable log of events
  • New events are always appended to the end of a partition
  • Each event gets a sequential offset (its position within the partition)
  • Partitions can live on different brokers in the cluster
Remember Order is guaranteed within a partition, but not across partitions. If you need strict ordering for a specific entity (e.g., all events for user-42), use a consistent key so they land in the same partition.

Partitioning Strategies

Round-robin (no key)

If no key is provided, Kafka distributes events across partitions in a round-robin fashion. This gives good load balancing but no ordering guarantee. 

# Round-robin — no key specified
producer.send(new ProducerRecord<>("orders", null, event));

Key-based partitioning

If a key is provided, Kafka hashes the key to determine the partition. All events with the same key go to the same partition, preserving order for that key. 

# Key-based — all events for order-123 land in the same partition
producer.send(new ProducerRecord<>("orders", "order-123", event));

Custom partitioner

You can implement your own partitioner for custom distribution logic — for example, pinning high-priority events to a dedicated partition.

⚠️ Warning Avoid creating too many partitions. Each partition adds overhead (file handles, memory). A good rule of thumb: start with partitions = max(3, number_of_brokers * 2) and monitor performance.

Replication

Each partition has replicas across multiple brokers for fault tolerance.

  • Leader — the replica that handles all reads and writes for a partition
  • Followers — replicas that stay in sync with the leader
  • If the leader broker fails, one follower becomes the new leader
# Describe a topic to see partition leadership
kafka-topics.sh --bootstrap-server localhost:9092 \
  --describe --topic orders

# Output:
# Topic: orders  Partition: 0  Leader: 1  Replicas: 1,2  ISR: 1,2
# Topic: orders  Partition: 1  Leader: 2  Replicas: 2,3  ISR: 2,3
# Topic: orders  Partition: 2  Leader: 3  Replicas: 3,1  ISR: 3,1

ISR = In-Sync Replicas. These are the followers that are fully caught up with the leader. If a follower falls too far behind, it's removed from the ISR set.

Key Takeaway

Partitions are your unit of parallelism. More partitions = more concurrent consumers, but also more operational overhead. Design your topics with a clear partitioning strategy based on your data and access patterns.
